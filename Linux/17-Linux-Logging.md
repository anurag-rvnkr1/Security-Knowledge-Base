# Introduction

Every action performed on a Linux system leaves traces.

These traces are called **logs**.

Logs record:

- User logins
- System startup
- Service activity
- Hardware events
- Application errors
- Security events
- Network events
- Authentication attempts

Linux logging is one of the most important skills for:

- Linux Administration
- System Engineering
- DevOps
- Site Reliability Engineering (SRE)
- Cybersecurity
- Digital Forensics
- Incident Response
- Threat Hunting

Without logs, troubleshooting and security investigations become extremely difficult.

# What is Logging?

Logging is the process of recording events generated by:

- Kernel
- Operating system
- Applications
- Services
- Security software
- Network components
- Users

Logs provide historical records that help administrators understand what happened on a system.

---

# Why Logging Matters

Logging enables organizations to:

- Troubleshoot failures
- Detect attacks
- Investigate incidents
- Meet compliance requirements
- Monitor applications
- Audit user activity
- Support forensic investigations
- Improve system reliability

# Syslog Facilities

Facilities identify the source of a message.

Common facilities:

| Facility | Purpose |
|-----------|----------|
| auth | Authentication |
| authpriv | Sensitive authentication messages |
| cron | Scheduled tasks |
| daemon | Background services |
| kern | Kernel messages |
| mail | Mail services |
| syslog | Syslog daemon |
| user | User-level applications |
| local0-local7 | Custom application logging |

| Level | Description |
|--------|-------------|

| Critical | Critical condition |

| Warning | Warning |
| Notice | Normal but significant event |

# `systemd-journald`

Most modern Linux distributions use:

```text
systemd-journald
```

Responsibilities:

- Collect logs from applications
- Collect kernel messages
- Capture boot information
- Store structured journal entries
- Provide searchable logs

---

# Advantages of Journald

- Structured log entries
- Fast searching
- Filtering by service
- Filtering by boot
- Filtering by priority
- Metadata support
- Integration with `systemd`

---

# Checking Journal Status

View journal service:

```bash
systemctl status systemd-journald
```

---

# Viewing Journal Entries

Display all logs:

```bash
journalctl
```

Newest entries:

```bash
journalctl -e
```

Latest entries:

```bash
journalctl -n 50
```

---

# Viewing Current Boot Logs

```bash
journalctl -b
```

Previous boot:

```bash
journalctl -b -1
```

Older boots can be referenced with `-2`, `-3`, and so on, if journal data is retained.

---

# Following Logs in Real Time

Equivalent to `tail -f`:

```bash
journalctl -f
```

---

# Filter by Service

Example:

```bash
journalctl -u ssh
```

Or:

```bash
journalctl -u sshd
```

The exact service name depends on the Linux distribution.

---

# Filter by Priority

Errors:

```bash
journalctl -p err
```

Warnings and above:

```bash
journalctl -p warning
```

Priority names include:

```text
emerg

alert

crit

err

warning

notice

info

debug
```

---

# Filter by Time

Since today:

```bash
journalctl --since today
```

Past hour:

```bash
journalctl --since "1 hour ago"
```

Specific time:

```bash
journalctl --since "2026-07-01 10:00:00"
```

Until a specific time:

```bash
journalctl --until "2026-07-01 12:00:00"
```

---

# Persistent Journal Storage

Check:

```bash
ls /var/log/journal
```

If the directory exists, journals are typically stored persistently across reboots.

If not configured for persistence, journals may reside under:

```text
/run/log/journal
```

which is usually volatile.

---

# Common Log Directory

Most Linux logs are stored in:

```text
/var/log
```

Display:

```bash
ls -lh /var/log
```

---

# Common Log Files

| Log File | Purpose |
|-----------|----------|
| `/var/log/messages` | General system messages (distribution-dependent) |
| `/var/log/syslog` | General system log (common on Debian/Ubuntu) |
| `/var/log/auth.log` | Authentication log (Debian/Ubuntu) |
| `/var/log/secure` | Authentication log (RHEL-family) |
| `/var/log/kern.log` | Kernel messages (where available) |
| `/var/log/boot.log` | Boot messages (distribution-dependent) |
| `/var/log/dmesg` | Boot-time kernel ring buffer output |
| `/var/log/cron` | Cron activity (RHEL-family) |
| `/var/log/maillog` | Mail service logs (RHEL-family) |

Different Linux distributions organize log files differently.

# Authentication Logs

Ubuntu/Debian:

```text
/var/log/auth.log
```

Red Hat Enterprise Linux:

```text
/var/log/secure
```

These logs commonly include:

- SSH logins
- Failed logins
- Successful logins
- `sudo` activity
- PAM events

---

# Viewing Authentication Events

Example:

```bash
sudo grep "Failed password" /var/log/auth.log
```

Or, using the journal:

```bash
journalctl -u ssh
```

Use the appropriate service name for your distribution.

---

# Viewing Login History

Successful logins:

```bash
last
```

Failed login attempts:

```bash
lastb
```

`lastb` requires the failed login database (`btmp`) and appropriate privileges.

---

# Log Entry Example

```text
Jul 21 10:30:15 web01 sshd[1456]: Accepted publickey for alice from 192.0.2.10 port 52000 ssh2
```

Typical components:

| Field | Meaning |
|---------|----------|
| Timestamp | Event time |
| Hostname | System generating the log |
| Process | Service name |
| PID | Process ID |
| Message | Event description |

---

# Structured Journal Metadata

Journal entries may include metadata such as:

- `_SYSTEMD_UNIT`
- `_PID`
- `_UID`
- `_HOSTNAME`
- `_COMM`
- `_EXE`

This enables precise filtering and analysis.

# Cybersecurity Perspective

Logs are among the most valuable sources of forensic evidence.

Security teams use logs to:

- Detect brute-force attacks
- Investigate unauthorized access
- Track privilege escalation
- Identify malware execution
- Reconstruct attack timelines
- Support incident response
- Meet audit and compliance requirements

Protecting log integrity is as important as collecting the logs themselves.

---

# Business Impact

Effective logging enables organizations to:

- Reduce troubleshooting time
- Improve service availability
- Detect security incidents earlier
- Demonstrate regulatory compliance
- Support root cause analysis
- Improve operational visibility

Without reliable logs, organizations may struggle to diagnose outages or investigate security events.

---

# Enterprise Best Practices

- Enable persistent logging where appropriate.
- Synchronize system time across hosts.
- Protect log files from unauthorized modification.
- Collect logs centrally for critical systems.
- Monitor authentication logs regularly.
- Define log retention policies based on business and regulatory requirements.
- Restrict access to sensitive logs.
- Regularly verify that logging services are functioning correctly.

---

# Key Takeaways

- Logging records operating system and application events.
- Modern Linux commonly uses `systemd-journald` with or without a syslog daemon.
- `/var/log` contains many important system logs.
- `journalctl` provides powerful filtering and search capabilities.
- Authentication and system logs are essential for administration and security investigations.
