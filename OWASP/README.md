# OWASP (Open Worldwide Application Security Project)

> "The world's largest non-profit organization dedicated to improving software security."

---

# Overview

The **Open Worldwide Application Security Project (OWASP)** is an international, non-profit foundation dedicated to improving the security of software applications. It provides free resources, methodologies, documentation, tools, standards, and educational materials that help organizations, developers, security engineers, penetration testers, DevSecOps teams, and students build secure software.

Unlike commercial security organizations, OWASP is entirely community-driven. Thousands of volunteers—including security researchers, developers, architects, and industry experts—collaborate to create and maintain high-quality security guidance that is freely available to everyone.

OWASP has become one of the most trusted authorities in application security. Today, governments, Fortune 500 companies, financial institutions, healthcare organizations, startups, universities, and security consultancies use OWASP standards to assess, develop, and improve application security.

Whether you're a beginner learning secure coding or an experienced penetration tester conducting enterprise security assessments, OWASP resources form a critical part of modern cybersecurity knowledge.

# History of OWASP

OWASP was founded in **2001** by **Mark Curphey** with the vision of making software security knowledge freely available to everyone.

Initially, application security resources were scattered across research papers, vendor documentation, and proprietary training materials. Developers often lacked accessible guidance on secure software development.

OWASP addressed this gap by creating an open community where security professionals could collaborate to produce practical resources for secure software development.

Over the years, OWASP has evolved into one of the most respected organizations in cybersecurity.

Today, the OWASP Foundation:

- Operates globally
- Hosts local chapters worldwide
- Organizes conferences and training events
- Maintains dozens of security projects
- Publishes internationally recognized standards
- Supports security education and research

# What is the OWASP Top 10?

The **OWASP Top 10** is one of the most influential cybersecurity awareness documents in the world.

It is a regularly updated report that identifies the **ten most critical security risks affecting modern web applications**.

The list is based on real-world vulnerability data collected from:

- Security consultancies
- Enterprise organizations
- Bug bounty programs
- Penetration testing firms
- Security vendors
- Open-source contributors

Millions of applications contribute anonymous vulnerability statistics that help OWASP identify the most common and impactful security weaknesses.

The OWASP Top 10 is not simply a "list of attacks." Instead, it serves as an awareness document that highlights the categories of vulnerabilities organizations should prioritize when designing, developing, testing, and maintaining software.

# How the OWASP Top 10 Is Created

Many beginners believe the OWASP Top 10 is simply a list of the "most dangerous attacks." In reality, it is a **data-driven awareness document** based on real vulnerability data collected from organizations worldwide.

OWASP collects anonymous vulnerability information from:

- Security consulting companies
- Penetration testing firms
- Fortune 500 organizations
- Cloud service providers
- Financial institutions
- Technology companies
- Government agencies
- Independent security researchers
- Bug bounty platforms
- Open-source security projects

Millions of vulnerability findings are analyzed to identify patterns and determine which categories of weaknesses are most common and impactful.

Rather than ranking individual vulnerabilities, OWASP groups them into **broad categories of security weaknesses**.

For example:

- SQL Injection
- NoSQL Injection
- LDAP Injection
- XPath Injection
- Command Injection

are all categorized under **A03: Injection** because they share similar root causes.

# Factors Considered

OWASP does not rank vulnerabilities based on popularity alone.

Several factors influence the final list, including:

## Frequency

How often the vulnerability appears during security assessments.

Example:

If thousands of applications contain Broken Access Control, it becomes a strong candidate for inclusion.

---

## Exploitability

How easy it is for an attacker to exploit the weakness.

Questions include:

- Can exploitation occur remotely?
- Does authentication need to be bypassed?
- Are specialized tools required?
- Is user interaction required?

---

## Technical Impact

What happens if exploitation succeeds?

Potential impacts include:

- Data theft
- Account takeover
- Remote code execution
- Privilege escalation
- Service disruption
- Complete application compromise

---

## Business Impact

Security incidents affect more than technology.

Business consequences include:

- Financial loss
- Regulatory fines
- Customer trust damage
- Legal action
- Operational downtime
- Brand reputation loss

# High-Level Overview of Each Category

## A01 – Broken Access Control

Occurs when users can perform actions or access resources beyond their intended permissions.

Examples:

- Accessing another user's profile
- Viewing confidential documents
- Changing another user's password
- Accessing admin functions without authorization

Common attacks:

- IDOR (Insecure Direct Object Reference)
- Privilege Escalation
- Forced Browsing

---

## A02 – Cryptographic Failures

Occurs when sensitive information is not adequately protected.

Examples:

- Plaintext password storage
- Weak encryption algorithms
- Missing HTTPS
- Poor key management

Often leads to:

- Credential theft
- Data breaches
- Identity theft

---

## A03 – Injection

One of the oldest and most dangerous vulnerability classes.

Occurs when untrusted input is interpreted as commands or queries.

Examples:

- SQL Injection
- NoSQL Injection
- Command Injection
- LDAP Injection
- XPath Injection

Potential consequences:

- Database compromise
- Remote command execution
- Data manipulation

---

## A04 – Insecure Design

Focuses on flaws in the application's architecture rather than coding mistakes.

Examples:

- Missing security requirements
- Poor threat modeling
- Insecure workflows
- Weak business logic

These vulnerabilities often cannot be fixed with a simple patch because they stem from fundamental design decisions.

---

## A05 – Security Misconfiguration

Results from insecure default settings or improper configuration.

Examples:

- Default credentials
- Directory listing enabled
- Debug mode in production
- Open cloud storage buckets
- Misconfigured HTTP headers

Misconfigurations remain one of the easiest attack vectors.

---

## A06 – Vulnerable and Outdated Components

Applications frequently depend on third-party libraries.

If those libraries contain known vulnerabilities, attackers can exploit them.

Examples:

- Outdated Log4j
- Vulnerable OpenSSL
- Old Spring Framework versions
- Deprecated JavaScript libraries

Supply-chain security has become increasingly important.

---

## A07 – Identification and Authentication Failures

Weak authentication mechanisms enable attackers to impersonate legitimate users.

Examples:

- Weak passwords
- Predictable session IDs
- Missing MFA
- Session fixation
- Credential stuffing

Compromised authentication often leads directly to account takeover.

---

## A08 – Software and Data Integrity Failures

Introduced in OWASP Top 10 (2021).

Focuses on software trust.

Examples:

- Insecure CI/CD pipelines
- Unsigned updates
- Malicious dependencies
- Supply-chain attacks
- Unsafe deserialization

The SolarWinds incident highlighted the importance of software integrity.

---

## A09 – Security Logging and Monitoring Failures

Organizations cannot respond to attacks they cannot detect.

Examples:

- Missing audit logs
- Poor log retention
- No alerting
- Missing failed-login monitoring
- Lack of incident response visibility

Proper logging is essential for threat detection and digital forensics.

---

## A10 – Server-Side Request Forgery (SSRF)

Occurs when an application fetches attacker-controlled URLs without proper validation.

Attackers may use SSRF to access:

- Internal servers
- Cloud metadata services
- Kubernetes APIs
- Redis
- Elasticsearch
- Internal admin panels

SSRF became increasingly important with the adoption of cloud infrastructure and microservices.

# Secure Software Development Lifecycle (Secure SDLC)

Building secure software is significantly more effective than attempting to secure software after deployment. The **Secure Software Development Lifecycle (Secure SDLC)** integrates security into every phase of software development.

Unlike the traditional SDLC, Secure SDLC ensures that security is considered from planning to maintenance.

Security activities include:

# OWASP Application Security Verification Standard (ASVS)

The **OWASP ASVS** is a comprehensive framework for verifying application security.

Instead of simply identifying vulnerabilities, ASVS defines **what secure applications should do**.

It provides detailed security requirements covering:

- Authentication
- Authorization
- Session Management
- Access Control
- Cryptography
- Error Handling
- Logging
- APIs
- File Uploads
- Business Logic
- Configuration
- Data Protection

### ASVS Levels

## Level 1

Basic security suitable for low-risk applications.

Example:

- Small business websites
- Internal portals

---

## Level 2

Standard security requirements for most enterprise applications.

Examples:

- Banking portals
- Healthcare systems
- SaaS platforms
- Government applications

---

## Level 3

Highest assurance level.

Used for:

- Critical infrastructure
- Defense
- Financial systems
- National security
- Highly sensitive environments
