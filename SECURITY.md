# Security Policy

Thank you for helping keep **Cybersecurity Notes** secure.

This repository is an educational knowledge base focused on defensive cybersecurity, secure engineering, and responsible security research. While it primarily contains documentation and reference material, we take the integrity of the repository and any supporting code or examples seriously.

---

# Supported Versions

The latest version of the repository is actively maintained.

| Version                  |    Supported   |
| ------------------------ | :------------: |
| Latest `main` branch     |        ✅       |
| Previous tagged releases | ⚠️ Best effort |
| Older archived versions  |        ❌       |

Please use the latest version whenever possible.

---

# Reporting a Security Vulnerability

If you discover a security issue related to this repository (for example, in sample code, scripts, workflows, or project infrastructure), **please report it privately** instead of opening a public GitHub Issue.

A good report should include:

* A clear description of the issue
* Steps to reproduce
* Expected behavior
* Actual behavior
* Impact assessment
* Suggested remediation (if known)
* Screenshots or logs (if relevant)

Please avoid including sensitive information such as credentials, tokens, API keys, or personal data.

---

# Responsible Disclosure

We ask all researchers to follow responsible disclosure practices.

Please:

* Allow reasonable time for the issue to be reviewed and addressed before public disclosure.
* Avoid publishing exploit details before a fix is available when the issue affects this repository.
* Report findings in good faith.
* Respect the privacy and security of others during research.

---

# Out of Scope

The following are generally considered out of scope for this repository:

* Typos or grammatical mistakes
* Broken external links
* Missing documentation
* Feature requests
* Vulnerabilities affecting third-party services that are unrelated to this project
* Issues requiring unauthorized access to external systems

---

# Response Process

When a valid security report is received, the general process is:

1. Acknowledge receipt.
2. Assess the reported issue.
3. Reproduce and validate the finding.
4. Determine severity and impact.
5. Develop and verify a fix.
6. Publish the fix.
7. Credit the reporter (with permission).

---

# Response Targets

While response times may vary depending on availability, the project aims for the following targets:

| Action                   | Target Time                    |
| ------------------------ | ------------------------------ |
| Initial acknowledgement  | Within 7 days                  |
| Initial assessment       | Within 14 days                 |
| Status updates           | As needed                      |
| Fix for confirmed issues | As soon as reasonably possible |

These targets are goals rather than guarantees.

---

# Security Best Practices

When contributing code or examples:

* Do not commit secrets, credentials, or API keys.
* Validate user input.
* Prefer secure defaults.
* Keep dependencies up to date.
* Avoid unsafe deserialization and command execution.
* Use parameterized queries where applicable.
* Follow the principle of least privilege.
* Remove unnecessary permissions and dependencies.

---

# Educational Content Disclaimer

This repository is intended for:

* Education
* Defensive security
* Authorized security testing
* Secure software development
* Security awareness

It is **not** intended to facilitate unauthorized access, illegal activity, or misuse of security techniques.

Readers are responsible for ensuring that any testing or experimentation is performed only on systems they own or are explicitly authorized to assess.

---

# Dependency Security

If this repository contains scripts or tools:

* Dependencies should be updated regularly.
* Security advisories should be reviewed before upgrading.
* Pull requests addressing confirmed dependency vulnerabilities are welcome.

---

# Safe Harbor

We appreciate responsible security research performed in good faith.

Nothing in this policy is intended to discourage legitimate security research conducted ethically and within the bounds of applicable laws and authorization.
