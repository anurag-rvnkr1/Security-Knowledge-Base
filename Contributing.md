# Contributing to Cybersecurity Notes

First off, thank you for your interest in contributing to **Cybersecurity Notes**! 🎉

Whether you're fixing a typo, improving an explanation, adding a new topic, or contributing a hands-on lab, your efforts help make this repository a valuable learning resource for the cybersecurity community.

---

# Our Goal

The goal of this repository is to provide:

* Accurate and practical cybersecurity knowledge
* Well-structured learning resources
* Clear explanations for beginners and professionals
* Ethical and defensive security guidance
* High-quality documentation that follows industry best practices

We welcome contributions that improve the educational value, technical accuracy, and usability of this project.

---

# Ways to Contribute

You can contribute by:

* Fixing spelling or grammar mistakes
* Improving technical explanations
* Updating outdated information
* Adding new cybersecurity topics
* Writing practical lab guides
* Creating diagrams or illustrations
* Improving formatting and readability
* Adding references to authoritative sources
* Reporting issues or suggesting improvements

---

# Before You Start

Please:

* Search existing Issues before creating a new one.
* Keep contributions focused on a single topic whenever possible.
* Ensure all information is technically accurate.
* Verify commands and configuration examples before submitting.
* Follow responsible disclosure practices when discussing security topics.

---

# Repository Structure

Please place new content in the appropriate directory.

Example:

```text
OWASP/
Networking/
Linux/
Windows/
Cloud-Security/
Containers/
Detection-Engineering/
Threat-Hunting/
SIEM/
Malware-Analysis/
Labs/
Cheatsheets/
Resources/
```

If your topic does not fit an existing category, open an Issue for discussion before creating a new top-level directory.

---

# Writing Guidelines

Please write documentation that is:

* Clear
* Accurate
* Practical
* Easy to understand
* Well formatted
* Grammatically correct

Prefer Markdown features such as:

* Headings
* Tables
* Lists
* Callout blocks
* Code blocks
* Images
* Diagrams

---

# Recommended Topic Template

Whenever possible, structure technical articles like this:

```markdown
# Topic

## Overview

## Why It Matters

## Architecture

## Key Concepts

## How It Works

## Practical Example

## Detection

## Prevention

## Best Practices

## Common Mistakes

## References
```

---

# Lab Contribution Guidelines

Practical labs should include:

* Objective
* Environment
* Prerequisites
* Steps
* Commands
* Expected Results
* Cleanup
* Lessons Learned

Do **not** include unauthorized testing instructions or content intended to facilitate attacks against systems without permission.

---

# Code Snippets

If adding scripts or code:

* Keep them simple and readable.
* Add comments where appropriate.
* Test before submitting.
* Avoid unnecessary dependencies.
* Include usage examples.

---

# Commit Message Guidelines

Use meaningful commit messages.

Examples:

```text
docs: add Linux privilege escalation notes

docs: improve Active Directory overview

docs: update OWASP Top 10 section

docs: add Kubernetes security checklist

fix: correct DNS explanation

style: improve Markdown formatting
```

Avoid messages like:

```text
update

changes

fixed

misc
```

---

# Pull Request Checklist

Before opening a Pull Request, ensure that:

* The content is accurate.
* Markdown renders correctly.
* Links work.
* Images display properly.
* Grammar has been reviewed.
* Formatting is consistent.
* Duplicate content has been avoided.

---

# Style Guide

Use:

* Descriptive headings
* Short paragraphs
* Bullet lists where appropriate
* Consistent capitalization
* Professional language

Prefer:

* "Cross-Site Scripting (XSS)" on first mention
* "Content Security Policy (CSP)" on first mention

Avoid unexplained abbreviations.

---

# References

Whenever possible, support technical information using authoritative sources such as:

* OWASP
* NIST
* CISA
* MITRE ATT&CK
* Microsoft Learn
* AWS Documentation
* Google Cloud Documentation
* Kubernetes Documentation
* Docker Documentation
* Mozilla Developer Network (MDN)

Clearly distinguish between official guidance and personal observations.

---

# Reporting Issues

If you discover:

* Broken links
* Incorrect information
* Outdated content
* Formatting issues

Please open a GitHub Issue with enough detail to reproduce or verify the problem.
